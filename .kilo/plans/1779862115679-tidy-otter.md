# Candidate Module Data Loading Optimization Plan

## Problem Analysis

The performance bottleneck is in `app/dashboard/page.tsx` which fetches ALL 4,000+ candidates in a single request without pagination or selective field fetching. This causes:
- Full table scan of candidates table (4,000+ records)
- Each record includes 22 fields + nested logs (potentially 10+ per candidate)
- Client-side filtering instead of database-level filtering
- Total payload size of ~5-10MB transferred and processed at once

### Current State
| File | Issue |
|------|-------|
| `app/dashboard/page.tsx:33-42` | Calls `/api/candidates` without `limit`, `essential`, or date filters - loads all 4,000 records |
| `app/api/candidates/route.ts` | Already supports `limit/offset`, `essential`, `countOnly` - but dashboard doesn't use them |
| `data/repositories/candidateRepository.ts` | Infrastructure exists but needs enhancements |

## Proposed Strategy: Three-Phase Optimization

### Phase 1: Dashboard - Implement Essential-Only + Aggregation Endpoint (High Priority)

**Changes to `app/dashboard/page.tsx`:**
1. Replace full candidate fetch with dedicated stats endpoint
2. Remove need for `includeLogs=true` - stats will be pre-computed server-side
3. Keep `essential=true` for any remaining candidate list needs (smaller fields)

**Changes to API layer:**
Create `/api/dashboard/stats` endpoint that returns pre-computed aggregates:
- Total counts by status
- Position distribution counts  
- Stage progression counts (derived from log aggregates)
- Date-based aggregations

**Net Result:** Eliminates need to transfer 4,000 full candidate objects; only aggregated stats are returned (~2-5KB payload)

### Phase 2: Enhance Repository - Position Distribution Aggregation (Medium Priority)

**Changes to `data/repositories/candidateRepository.ts`:**
1. Add `getDashboardStats()` function with all server-side aggregations
2. Returns pre-computed values: total, today, lastWeek, lastMonth counts, statusCounts, positionDistribution, stageTotals

**Dashboard stats SQL logic:**
```sql
-- Total count
SELECT COUNT(*) as total FROM candidates;

-- Today/Last Week/Last Month with date filtering
SELECT COUNT(*) FROM candidates WHERE date_applied >= date('now');

-- Status counts
SELECT s.name as status, COUNT(*) as count FROM candidates c JOIN statuses s ON c.status_id = s.id GROUP BY s.id;

-- Position distribution (top 8)
SELECT p.name as position, COUNT(*) as count FROM candidates c JOIN positions p ON c.position_id = p.id GROUP BY p.id ORDER BY count DESC LIMIT 8;

-- Stage progression (from activity_logs)  
-- For each candidate, determine which stages they've been in based on their log history
-- This requires checking if each status exists in their logs
```

**Stage calculation logic:**
- Applied: candidates with logs but not Shortlisted/Not Suitable in logs
- Shortlisted: has 'Shortlisted' in logs
- Not Suitable: has 'Not Suitable' but not Shortlisted in logs

### Phase 3: Smart Caching for Stats Endpoint (Low Priority)

**Changes to `app/api/dashboard/stats/route.ts`:**
1. Add in-memory cache for stats data (5-minute TTL)
2. Implement `stale-while-revalidate` pattern - serve cached data immediately while fetching fresh data in background

This is optional since SQLite queries are fast enough for aggregated data.

## Implementation Order

1. **Phase 1** (Immediate - 1-2 hours): Create `/api/dashboard/stats` endpoint and refactor dashboard to use it
2. **Phase 2** (Follow-up - 1-2 hours): Add optimized helper functions to repository for aggregations
3. **Phase 3** (Optional - 1 hour): Add caching layer for stats endpoint

## Specific Code Changes

### 1. Dashboard Fix (`app/dashboard/page.tsx`)

**Before (lines 27-45):**
```javascript
const fetchData = async () => {
  setLoading(true);
  const params = new URLSearchParams();
  if (startDate) params.set('startDate', startDate);
  if (endDate) params.set('endDate', endDate);
  if (owner) params.set('owner', owner);
  params.set('includeLogs', 'true');
  
  const [candRes, recRes] = await Promise.all([
    fetch(`/api/candidates?${params}`),
    // ...
  ]);
};
```

**After:**
```javascript
const fetchData = async () => {
  setLoading(true);
  
  // Use dedicated stats endpoint for aggregated data
  const [statsRes, recRes] = await Promise.all([
    fetch(`/api/dashboard/stats?${new URLSearchParams({ startDate, endDate, owner })}`),
    // ...
  ]);
  const stats = await statsRes.json();
  // Process pre-aggregated stats instead of full candidate objects
};
```

### 2. API Enhancement - Default Limit (`app/api/candidates/route.ts`)

Add default limit to prevent accidental full-table fetches:
```javascript
const limitParam = searchParams.get('limit');
const limit = limitParam ? Number(limitParam) : 25;  // Default limit
```

### 3. New Aggregation Endpoint (`app/api/dashboard/stats/route.ts`)

Create dedicated endpoint returning pre-computed aggregates:
- Total counts by status
- Position distribution
- Stage progression counts
- Date-based aggregations

This eliminates the need to transfer full candidate objects for dashboard analytics.
