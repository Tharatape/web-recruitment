# Database Relationship Analysis & Reset Plan

## Current Database Schema

### Tables and Relationships

```
statuses (id, name)
    ↑
    │
    ├── candidates.status_id → statuses(id)
    ├── activity_logs.status_id → statuses(id)
    └── (STATUSES: Applied, Not Suitable, Shortlisted, 1st Interview, 2nd Interview, Not Selected, Selected, Offer Accepted, Offer Declined, Hired, Not Hired)

owners (id, name)
    ↑
    │
    ├── candidates.recruiter_id → owners(id)
    ├── activity_logs.recruiter_id → owners(id)
    └── (OWNERS: Sarah Mitchell, James Carter, Emily Davis, Michael Brooks)

positions (id, name)
    ↑
    │
    ├── candidates.position_id → positions(id)
    └── (POSITIONS: Sales Executive, Marketing Specialist, Software Engineer, Data Analyst, HR Manager, Financial Analyst, Customer Service, Project Manager, Business Analyst, Operations Manager)

candidates (id, unique_id, name, phone, nid, email, position_id, experience, experience_level, date_applied, status_id, recruiter_id, age, weight, height, bmi, expected_salary, education, address, language, license, previous_employment, ai_summary, type, department, degree, major, toeic)
    ↑
    │
    └── activity_logs.candidate_id → candidates(id) ON DELETE CASCADE

jds (id, name, position, created_at, disabled)
    ↑
    │
    └── jd_checklists.jd_id → jds(id) ON DELETE CASCADE

jd_checklists (id, jd_id, category, criterion_order, criterion_text)
```

## Relationship Summary

1. **candidates** references **positions**, **statuses**, and **owners** (recruiter) via foreign keys
2. **activity_logs** references **candidates** (with CASCADE delete), **statuses**, and **owners**
3. **jd_checklists** references **jds** (with CASCADE delete)
4. All relationships use foreign key constraints
