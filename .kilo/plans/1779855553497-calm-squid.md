# Performance Optimization Plan: Pagination and Lazy Loading

## Problem Analysis
Both `/app/matching/page.tsx` and `/app/applications/page.tsx` fetch ALL 4,000+ candidates at once via `/api/candidates` without using the API's built-in pagination (`limit`/`offset`) parameters. Client-side pagination slices already-loaded data, causing initial latency. The expanded row view also renders all logs, ai_summary, and other heavy fields upfront.

## Solution: Server-Side Pagination + Lazy Loading for Expanded Data

### Phase 1: Backend Changes

#### 1. Add Essential-Felds Query Function
**File: `data/repositories/candidateRepository.ts`**

```typescript
// Essential fields for table display (without heavy fields like ai_summary, logs, etc.)
export function getCandidatesEssentialOnly(filters: {
  startDate?: string;
  endDate?: string;
  owner?: string | null;
  search?: string;
  position?: string[];
  status?: string[];
  limit?: number;
  offset?: number;
}): DbCandidateEssential[] {}
```

`DbCandidateEssential` interface will have fields: `id`, `unique_id`, `name`, `phone`, `nid`, `email`, `position`, `experience`, `date_applied`, `status`, `recruiter`

- Add `getCandidateFullById(id)` returning complete candidate with logs

#### 2. Update API Endpoint
**File: `app/api/candidates/route.ts`**
- Add `essential=true` param → calls `getCandidatesEssentialOnly()`
- Add `countOnly=true` param → returns `{ total: number }` for pagination
- Default behavior unchanged (returns all fields) for backward compatibility
- Add optional filter params to match frontend filters (startDate, endDate, owner, search, status, position)

### Phase 2: Frontend Changes

#### 3. Refactor Matching Page
**File: `app/matching/page.tsx`**

State changes:
- Remove single `candidates` state
- Add `paginatedCandidates` for current page data
- Add `fullCandidates` Map<string, DbCandidate> to cache expanded candidates
- Add `expandedCandidateData` state for currently expanded candidate details

Data fetching:
```typescript
// Initial load
fetch(`/api/candidates?limit=25&offset=0&essential=true`)

// When page changes
fetch(`/api/candidates?limit=${pageSize}&offset=${(page-1)*pageSize}&essential=true`)

// When expanding row
fetch(`/api/candidates/${uniqueId}/full`)
```

UI changes:
- Show loading state in expanded row while fetching full data
- Cache full data to avoid re-fetching on re-expansion

#### 4. Refactor Applications Page
**File: `app/applications/page.tsx`**
- Same changes as matching page for consistency

### Phase 3: Optional Enhancements
- Add loading skeletons for expanded rows
- Preload next page on hover

## Technical Considerations
- The existing `getCandidatesWithFilters` already supports `limit`/`offset` - just need to add `essential` param
- Server-side filtering will be implemented to reduce data transfer
- Need to add `expMin`/`expMax` filter params to API (missing currently)
- Sorting will be done client-side on current page data (maintaining current UX)
- **Selected approach**: Server-side filtering with server-side pagination + lazy loading

## Implementation Notes
- `getCandidatesEssentialOnly` will include experience in the SELECT to support expMin/expMax filtering
- API will accept filter params and apply them before pagination to get accurate page counts
- Add `expMin`/`expMax` filter params to API (missing currently)

## Phase 4: Testing Verification
- Verify pagination controls work correctly
- Verify filter counts are accurate with server-side filtering
- Verify expand/collapse lazy loading works
- Verify no regression in existing functionality
